コンパス(#コンパス戦闘摂理解析システム)というゲームで、プレイヤー同士の対戦バトルのマッチング表をランダムに割り振るhtmlプログラムを生成してください。

ファイル構成は以下の通りです。
・CompassCustomBattle.html　メイン処理
・List_Hero.js　使用キャラクター(ヒーロー)の一覧を定義した定数ファイル。オブジェクトのプロパティ構成は「ロール定数、ヒーロー定義名、ヒーロー和名」とする。(ヒーロー定数である、アタッカー、ガンナー、スプリンター、タンクの4種はメイン処理側で定義)
　サンプル：const LIST_HERO = [ { role: 'ATTACKER', name: 'ATTACKER_001', jp_name: 'ヒーロー和名A' } ];
・List_Stage.js　対戦ステージの一覧を定義した定数ファイル。オブジェクトのプロパティ構成は「対戦人数、ステージ定義名、ステージ和名」とする。対戦人数は「3対3」と「2対2」の2パターンのみ。
　サンプル：const LIST_STAGE = [ { mode: 2, name: 'stage_2_001', jp_name: 'ステージ和名B' } ];
・List_NotOwned.js　参加者ごとに使用不可能なヒーローの一覧を定義した定数ファイル。オブジェクトのプロパティ構成は「参加者和名、ヒーロー定義名」とする。
　サンプル：const LIST_NOT_OWNED = [ { player_jp_name: 'ダミーC', hero_name: 'TANK_009' } ];

なお、各List_XXX.jsファイルの具体的な内容は私が手打ちで入力するので、これらのファイルの生成は不要です。
各List_XXX.jsを編集することで、CompassCustomBattle.htmlを修正することなく挙動をコントロールできるように作成してください。

詳細仕様は以下の通りです。

カスタムバトル対戦表 システム仕様書
1. アプリケーション概要
目的: チーム制対戦ゲームにおける、参加プレイヤーのランダムなチーム分け、使用キャラクター（ロール）の割り当て、および対戦ステージの自動抽選を行う。

動作環境: Webブラウザ（PC・スマートフォン両対応）。外部ライブラリを一切使用しない1ファイルのHTMLで完結しており、完全オフライン環境でも動作する。

2. 画面・UI仕様
テーマ: ダークモード基調。

全体背景: #121212 / カード背景: #1e1e1e / 文字色: #e0e0e0

チームカラー: 紅チーム（#e83828）、蒼チーム（#004dff）

レイアウト: シングルページ構成（最大幅600pxのレスポンシブデザイン）。上部に「設定・入力エリア」、下部に「対戦表（結果）エリア」を配置。

3. データ定義

3.1 キャラクター・カテゴリリスト
4つのカテゴリ（ロール）と、それぞれに属するキャラクター群。

アタッカー、ガンナー、スプリンター、タンク

3.2 ステージリスト
対戦人数モードに応じた専用のステージ群。

2 vs 2用、3 vs 3用

4. 機能仕様
4.1 メンバー管理機能
追加: テキストボックスに名前を入力し、「追加」ボタンまたは Enter キー押下で登録。

表示・削除: 登録されたメンバーはタグ（バッジ）形式で一覧表示され、横の「×」ボタンで個別に削除可能。

データ永続化: メンバーの追加・削除が行われるたびに、ブラウザの localStorageに自動保存され、次回アクセス時に復元される。

XSS対策: メンバー名の画面描画時は .innerHTML ではなく .textContent を使用し、悪意のあるスクリプトタグ等が入力されても文字列として安全に処理（サニタイズ）する。

4.2 抽選設定オプション
モード選択: 2 vs 2 モード または 3 vs 3 モード をトグルボタンで選択。
デフォルトは 2 vs 2 モードとする。

カテゴリのみ表示: チェックONの場合、抽選結果のキャラクター名（ヒーロー和名）を隠し、カテゴリ名（ロール）のみを出力する。

ロール固定機能：有効化すると、どのロールが必ず選出されるか決定できる。4つのロールからチェックボックスで有効化し、有効化できるのは選択したモードの1チームあたりの人数（2人または3人）まで。
有効化したときのみ、対象ロールのチェックボックスUIを縦並び（アタッカー、ガンナー、スプリンター、タンクの順）で表示する。

同ロール選出機能：ロール固定機能と排他選択とする（両方同時に有効化できない）。有効化すると、2チームとも同じロールの組み合わせが選出されるようになる。

キャラクターの選出対象除外機能：キャラクターが抽選対象から除外できるようにする機能。キャラクター除外ボタンを押すと、キャラクターリストを元にした一覧(ロールごとに並べて表示される)画面がポップアップし、チェックしたキャラクターを除外できる。ロールごとで一括チェック機能も追加する。
　なお、この除外オプションはロール固定や同ロール固定など他の抽選オプションの有効・無効に関わらず動作するものとする。

  キャラクターの選出対象除外機能（詳細仕様）
  ボタン押下で表示されるモーダルウィンドウは、以下のUI構造を厳守すること。
  モーダル全体の構造:
    画面全体を覆う半透明のオーバーレイ（background: rgba(0,0,0,0.8); z-index: 1000;）を配置し、中央にメインウィンドウを表示する。
    ウィンドウサイズは width: 90%; max-width: 600px; max-height: 80vh; とし、内部でスクロール可能（overflow-y: auto;）にすること。
    背景色は #1e1e1e、枠線は 1px solid #333、角丸は 8px とする。
  ヘッダー（固定表示）:
    ウィンドウ上部にタイトルと「閉じる」ボタンを配置。
    閉じるボタンは position: sticky; top: 0; またはヘッダー内に固定し、スクロールしても常に右上に見えている状態にすること。
    閉じるボタンのスタイル：background: #333; color: #fff; padding: 5px 15px; border-radius: 4px;
  キャラクター表示エリア（グリッドレイアウト）:
    ロール（アタッカー等）ごとにセクションを分ける。
    各ロールの冒頭に、そのロールの全キャラを一括で選択/解除できる「全選択チェックボックス」とロール名を配置する。
    キャラクター一覧は、CSS Grid (display: grid;) を使用して配置する。
    カラム設定：grid-template-columns: repeat(auto-fill, minmax(130px, 1fr));（これにより3〜4列で綺麗に並ぶ）。
    各要素のサイズ：1人分の表示エリアを height: 40px; 程度の固定サイズとし、枠線で囲む。
    テキスト処理：名前が長い場合は font-size: 0.8rem; に縮小し、一行に収めること。
  インタラクション:
    チェックボックス自体をタップしなくても、キャラクター名が書かれた枠内（label要素）のどこをタップしてもチェックが切り替わるように、input と label を適切に紐付ける。
    チェックが入った（除外された）キャラクターは、背景色を #333 に変更し、テキストをグレーアウトさせることで「無効化されていること」を視覚的に分かりやすくする。

4.3 チーム生成・抽選ロジック（「チーム設定」ボタン）
バリデーション: 登録メンバー数が、選択モードの必要人数（4名または6名）を満たしていない場合はアラートを出し、処理を中断する。

ステージ抽選: 選択モードに応じたステージリストから、ランダムに1つを選出する。

プレイヤー選出: 登録メンバー全体をシャッフルし、必要人数分を抽出し、選出されたメンバーの『過去のロール履歴』に基づき、全体のペナルティが最小になるように枠を割り当てる

枠（ロール）の割り当て:

前提条件：ロール固定の有無や同ロール選出の有無など、いかなる条件に関わらず、同チーム内で同ロールが選ばれることは絶対にないようにする（各チーム各ロール最大1枠）。

通常ルール（ロール固定無し）:

アタッカー、ガンナー、スプリンター、タンクの中から、対戦人数に応じて3枠、または2枠を重複なしでランダムに選定する。
選出されたプレイヤー（2名または3名）を、この選定された枠のいずれかにランダムに割り当てる。

特殊ルール（ロール固定有効）: 
固定されたロールの枠は固定で用意し、固定数が対戦人数より少ない場合は残りのロールを余った枠に重複なしでランダムで割り当てる。
選出されたプレイヤー（2名または3名）を、この選定された枠のいずれかにランダムに割り当てる。

特殊ルール（同ロール選出有効）:

アタッカー、ガンナー、スプリンター、タンクの中から、対戦人数に応じて3枠、または2枠を重複なしでランダムに選定し、2チームとも同じロールの組み合わせにする。
選出されたプレイヤー（2名または3名）を、この選定された枠のいずれかにランダムに割り当てる。

キャラクター選出: 用意された枠のカテゴリに応じ、各キャラクターリストからランダムに1名を選出する。その際、List_NotOwned.jsに名前のある参加者がプレイヤーにいる場合はその一覧を参照し、一覧にあるキャラクターをそのプレイヤーの抽選対象から外す。

4.4 重複対策
完全にランダムにした場合、同じメンバーで構成されたチームや、同じキャラクターを使用し続ける人、同じロールになり続ける人、連続で不参加になる人のような偏りが生じるため、内部（メモリ上またはlocalStorage）に過去のマッチング履歴状態を保持し、以下の対策ロジックを実装する。

・参加者の選出（待機スコア制）
メンバーごとに「待機スコア（初期値0）」を管理する。試合が行われるたび、その試合に『参加した』メンバーのスコアを0にリセットし、『参加しなかった』メンバーのスコアを+1する。抽選時は待機スコアが高いメンバーを優先して選出し、同点の場合はランダムとする。これにより連続不参加を防ぐ。

・チーム分け（アンチ・ペアリング）
選出されたメンバーを紅・蒼に分ける際、過去3試合の「同じチームになった組み合わせ履歴」を参照する。考えられる全チーム分けパターンに対し、過去3試合で同じチームだったペアが含まれるごとにペナルティ点（+1点）を加算し、ペナルティ点が最も低いパターンの集合の中からランダムにチーム分けを採用する。これにより特定のメンバー同士が固まるのを防ぐ。

・ロールの割り当て
各プレイヤーごとに「直近2試合で担当したロール」を履歴として保持する。選定されたロール枠をプレイヤーに割り当てる際、可能な限り直近2試合のロールと被らないパターンを探索して適用する。条件が厳しく割り当て不可能な場合のみ、制限を解除してランダムに割り当てる。

・キャラクターの割り当て（LRU方式）
各プレイヤーごとに「過去10試合のヒーロー使用履歴」を保持する。各ロールの抽選対象ヒーローから、「使用履歴に存在しない（まだ使っていない）ヒーロー」を優先してランダムに選ぶ。すべて使用済みの場合は、履歴の中で「最も古い試合で使用したヒーロー」から選出する。なお、List_NotOwned.jsによる使用不可能ヒーローが定義されている参加者については、常に抽選対象から除外した上でこの履歴判定と割り当てを行う。

4.5 結果表示
抽選結果を画面下部のカードUIに描画する。
対戦表は「参加者名」「ロール」「ヒーロー和名」の順に表示し、各カラムの位置が縦で綺麗に揃うようにグリッドレイアウト等で整える。
表示順序はチームごとに「アタッカー → ガンナー → スプリンター → タンク」の順でソートする。
参加者名やヒーロー和名が長く表示領域を超える場合は、自動的にフォントサイズを縮小して枠内に収める。
チームごとの参加者リストだけ、文字列としてコピーできるボタンを追加する。ボタンを押下した際はメッセージボックスではなく、ボタン自体の表示テキストと色を一定時間変化させて成功を通知する。
参加者リストのフォーマットは以下の通りとする。
紅チーム：参加者1、参加者2...
蒼チーム：参加者1、参加者2...

5. デバッグ機能
5.1 統計機能
関数名：dumpHistory
内部処理を確認するため、画面上に出す必要はないが、特定のコマンドを開発者モードで実行することでこれまでの参加者リストを全て(このデータはlocalStorageなどで永続化する必要はない)テキスト出力する機能を追加 
出力形式はExcel等へ貼り付けやすいようにタブ区切り（TSV形式）とし、コンソールログへの出力をデフォルトとする。
フォーマットは
マッチNo \t 所属チーム \t 参加者 \t 使用ロール(和名) \t 使用ヒーロー(和名)
とする。そのバトルの参加者分の行が記録されるものとする。

5.2 動作テスト機能
関数名：checkMatching
抽選機能の動作確認のため、現在の設定でマッチング処理を自動で100回実行(画面上のUIでボタンを押したのと同じ処理。もし画面再描画などで高負荷が予想される場合、このループ中は再描画を行わない)する。
